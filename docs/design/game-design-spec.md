## 一、项目概述

### 1.1 游戏定位

面向**成人及青年玩家（16+）**、对思维方式本身感兴趣的认知解谜游戏。玩家扮演寓言旅者"莱米"（统一手绘墨线小兔子，无年龄无性别的象征形象），驾驶飞船"流辉美慧号"穿越星海，修复 **33 颗**故障星星——但修复不是目的，而是**提炼**：把玩家生活中早已有的模糊思维直觉，唤起 → 命名 → 整理为 Ta 能随身调用的认知工具。游戏结束时，玩家带走的是一本 33 张的认知工具册，不是通关记录。本游戏不做儿童产品——认知工具对儿童过于抽象；但它也不刻意冷峻，童话意象受欢迎（《小王子》式温度），详见 §4.2《小王子》测试法。

**为何是 33 关而非 50 关**：原始 50 工具清单在"思维工具定义层"（C 轴）可以互不相同，但按玩家视角的双轴正交性——**A 轴 = 手上动作，B 轴 = 系统阅读方式**——严苛审视，真正能给玩家带来新体验的关卡是 33 个。剩下 17 个要么 A+B 与已有关共享（只是换了工具标签），要么无法在 5 分钟交互内落地、必须靠旁白文字说教。详见《二、2.5 从 50 到 33：正交性裁定》与 `docs/design/stage-tonality-verification.md`。

### 1.2 核心体验

- **修复即提炼**：玩家在生活中早已有各种模糊的思维直觉（比如"分过类""倒推过""做过权衡"），但大多数没有被命名和整理。每颗星星的故障对应一种认知动作，修复过程**唤起 → 命名 → 整理**玩家的既有直觉，把它从模糊经验提炼为清晰的认知工具，进入玩家的"**认知工具册**"（见 §6.5）。这不是从零教学，而是**把玩家已经会的事情变成 Ta 能命名、能复用的工具**。
- **无文字压力**：核心交互靠视觉和操作，文字提示为辅助；工具的"命名"发生在关卡通关之后，不打断过程中的沉浸。
- **渐进复杂度**：5 个阶段从具象操作逐步过渡到抽象思维（习得 → 组合 → 超越 → 内观 → 生成）。
- **带走的礼物**：智慧结晶收集系统 + 认知工具册 index card——每张卡片有"**何时使用 / 生活例子**"面，让玩家能在游戏之外真的调用这些工具。游戏结束时，玩家带走的不是一个通关记录，而是一本随身可用的认知工具册。

| 项目 | 选择 |
|------|------|

## 二、五阶段认知进阶体系（全局结构）

### 2.1 阶段总览

| 阶段 | 名称 | 关卡 | 关数 | 认知主题 | 交互复杂度 | 视觉风格演进 |
|------|------|------|------|---------|-----------|-------------|
| 1 | 秩序之基 | M01-M10 | **10** | 基础思维工具 | 拖拽、点击、滑块 | 具象、明亮、机械感 |
| 2 | 逻辑之网 | M11-M16 | **6** | 组合工具解决复合问题 | 组合操作、多步骤 | 半抽象、暖色调 |
| 3 | 系统之舞 | M17-M23 | **7** | 动态复杂性 | 动态系统、实时交互 | 抽象、流动、冷暖交织 |
| 4 | 认知之镜 | M24-M28 | **5** | 元认知 | 间接控制、多视角切换 | 高度抽象、光影为主 |
| 5 | 演化之巅 | M29-M33 | **5** | 自我超越 | 规则重构、开放式 | 极简 + 万花筒 |
|   | **合计** |      | **33** |          |           |              |

**金字塔分布的设计意图**：5 个阶段按 10-6-7-5-5 的梯度逐级收窄（Stage 3 因为吸收了反馈循环而比 Stage 2 多 1 关），**顶端最窄** = Stage 5 每一关都是里程碑，不是"又十关套路"。

### 2.2 阶段叙事线与认知弧

每个阶段由"**目标（认知动词）+ 角色（演进叙事）**"双线承载，两线同步演进才构成"思维生长"的调性。阶段之间是**质变**而非难度加深——玩家做的事情的**种类**在变，不是"更难的同一件事"。

| 阶段 | 目标（认知动词） | 角色（演进叙事） |
|------|----------------|----------------|
| **1 秩序之基**（M01-M10, 10 关） | 习得处理信息与解决问题的**基础思维动作**——从具象操作中把工具握在手里 | 星空**维修工**——让失序的事物恢复秩序与和谐 |
| **2 逻辑之网**（M11-M16, 6 关） | 把基础动作**组合应用**，诊断复合系统的故障、设计多步修复方案 | 星空**工程师**——攻克需要推理与设计的复合难题 |
| **3 系统之舞**（M17-M23, 7 关） | **超越静态修复**，在动态、不确定、多主体互动的系统中借力做决策 | 星空**导航家**——于湍流中行舟，与系统共舞而非与之对抗 |
| **4 认知之镜**（M24-M28, 5 关） | **把目光从外部世界收回**，审视并优化自己的认知过程与心智模型 | 星空**哲学家**——理解认知的建构性、局限性与可塑性 |
| **5 演化之巅**（M29-M33, 5 关） | 挑战认知的终极边界，练习**范式转换、意义创造、认知演化**的元能力 | 星空**建筑师**——重新定义问题、创造新规则、设计自身认知的演化 |

**动词弧（调性主心骨，不可漂移）**：

```
习得 → 组合 → 超越 → 内观 → 生成
 (手)    (配合)   (取舍)   (自照)   (创造)
```

每关设计时必须能回答："本关训练的认知动词，属于动词弧上的哪一段？是否与所在阶段的**种类**一致（而非只是更难的同类动作）？"若一关训练的动词与所在阶段不匹配，要么换关卡，要么换阶段——不允许"氛围像这一阶段但动词属于另一阶段"的漂移。

### 2.3 33 关完整索引

以下 33 关是按"玩家 A+B 正交性"（A 轴 = 手上动作，B 轴 = 系统阅读方式）从原始 50 工具清单裁定后的终版。**关卡 ID 已重新编号为 M01-M33 连续编号**。裁定过程见 `docs/design/tool-dedup-audit.md`（删除/合并清单）+ `docs/design/50-puzzle-comparison.md`（原始 50 关定稿）；原始 50 关 ID → 新 33 关 ID 没有单独整理成映射表，而是通过重编号直接废弃旧 ID。`docs/design/stage-tonality-verification.md` 是去重前的 50-关调性审计历史档案，仍用 50-关编号，**不是映射表**。

### 2.4 智慧结晶写作准则

每关结束时的一句智慧结晶是本游戏的"最小文学单位"，既是玩家的认知战利品，也是认知工具册的文字部分（见 §6.5）。所有 33 关的结晶文案都必须遵守以下六条准则，避免漂移回说教或幼儿口吻：

- **不向年龄稀释**：同一句话既要能被孩子做直觉理解（家、路、纸的意象），又要能被成人读出认知学意味（范畴化、校准、逆推）。若读起来像绘本旁白，就废稿重写。
- **两段式结构**：前半否定旧直觉或陈述表象，后半给出新直觉或本质——用破折号 / 冒号 / 分号衔接。范例："稳定不是静止，而是无数次及时的微小回应。"
- **具体动词与意象**：优先使用可视化动词（归处、倒走、相遇、移山）和具象名词（链条、接缝、两条边），避免抽象术语堆砌。
- **一句一断**：字数控制在 20-45 字之间，能让玩家一眼读完、一口默念；不写两句、不写段落。
- **非说教语气**：禁用"要…才行 / 记住 / 不要慌 / 小朋友"等直接教导用语；让领悟发生在读者自己心里。
- **与思维工具一一对应**：每句必须与该关 `cognitiveSkill` 构成"隐喻-工具"的精准映射，不得退化为一般性心灵鸡汤。写完后做一次对照校验：遮住工具名，读者能否从这一句反推出该关要训练的思维动作？

## 四、美术风格体系

### 4.1 统一手绘语言

游戏整体采用**统一的 Arrog 式简约手绘语言**，不再拆成"角色水彩 / 世界墨线"两套媒介。主角、场景、机械、界面、背景工具全部使用**同一种清晰可见的手绘墨线轮廓**，再辅以低饱和、克制的手工淡彩。画面首先成立于线条，颜色只服务线条、结构与情绪，不得反过来把线条洗掉。

| 维度 | 规范 | 服务目标 |
|------|------|---------|
| **线条** | **清晰可见的手绘墨线轮廓**：轻微抖动、粗细变化、不规则手工感 | 让所有对象都先以"被画出来"的方式成立，而不是靠上色成立 |
| **色彩** | **低饱和手工淡彩**：像旧纸上的克制点染，不追求饱满覆盖 | 维持 Arrog 式朴素、干燥、安静的气质 |
| **结构** | **功能性简约机械**：设备真实可理解，但部件少、结构清楚、无装饰堆砌 | 保证谜题装置既像能工作，又不陷入复杂炫技 |
| **构图** | **大量留白 + 舞台感**：主物件突出，背景只给少量必要线索 | 让玩家一眼看见当下的认知动作焦点 |

**固定配色体系**（全项目统一调用）：
米白 · 灰蓝 · 灰绿 · 陶土红 · 赭石黄 · 暗橄榄 · 木炭黑 · 少量黄铜暖色

**配色原则**：
- 米白、灰蓝、灰绿作为大面积底色与空间气氛色
- 陶土红、赭石黄、暗橄榄作为局部识别色与情绪点染
- 木炭黑只用于线条、切口、阴影边缘和结构强调
- 黄铜暖色只点到机械关键部位，不可大面积铺金属高光
- 禁止鲜艳糖果色、霓虹色、紫色主导和厚重饱和渐变

**核心区分**：向 Arrog 借的不是某个单独造型，而是**"线条永远清晰存在"**这一美术底盘。允许轻微淡彩、纸张质感和局部识别色，但绝不能把画面做成软糊糊的水彩氛围插图，也不能做成繁复机械插画。线条必须从角色到设备到背景始终统一。

### 4.1.1 线条、色彩与结构的硬规则

- **线条优先于上色**：任何对象在去掉颜色后也必须能靠线条成立
- **颜色只是辅助**：颜色只能轻轻点染，不允许把手绘轮廓冲淡
- **角色 / 设备 / 背景统一语言**：不能出现角色软、设备硬、背景糊的媒介割裂
- **装饰性元素尽量删掉**：优先保留功能部件，删除解释性花纹、无用装饰和炫技小零件
- **机械结构必须可信**：设备可以简约，但必须一眼看懂为什么能运作，不能沦为抽象符号
- **背景只给少量机械线索**：背景可以有支架、吊臂、工作台、杆件等，但只能作为陪衬，不抢主装置
- **留白是正式设计手段**：空白区域不是没画，而是用于压住噪音、突出认知动作

### 4.1.2 阶段视觉演进（统一语言内的情绪渐变）

风格不切换（线条语言、简约结构、留白方法全游戏一致），但**色温、意象密度、抽象度**按阶段渐变，形成"从具象机械到抽象规则"的情绪弧：

| 阶段 | 主导色温 | 意象类型 | 抽象度 | 主导情绪 |
|------|---------|---------|-------|---------|
| 1 秩序之基 | 偏暖（米白 / 陶土红 / 赭石黄 / 少量黄铜） | 具象单体机械（齿轮、罗盘、分拣盘） | 低 | 安静、秩序、可理解 |
| 2 逻辑之网 | 暖中带冷（加入灰蓝、灰绿、暗橄榄） | 简化的组合机械与路径结构 | 中 | 观察、推敲、联结 |
| 3 系统之舞 | 冷暖并存 | 半具象场域、平衡结构、流动关系 | 中 | 张力、借力、动态 |
| 4 认知之镜 | 偏冷（灰蓝 / 灰绿 / 木炭黑为主） | 光影、反射、框架、镜面 | 高 | 内省、安静、距离感 |
| 5 演化之巅 | 极简对比（少量识别色 + 大量留白） | 规则、几何、边界、抽象装置 | 极高 | 自由、超越、生成 |

**注**：阶段差异体现在"**同一支手绘笔，画不同浓度的世界**"，不是换画风。越到后期，越应减少无关装饰，让结构、线条和认知动作自己说话。

### 4.2 童话作为成人艺术语言：《小王子》测试法

本游戏**不去童话化**——小兔子、花粉、彩虹、种子这类意象是允许且鼓励的。但每个意象必须通过"《小王子》测试"，以免滑向儿童化稀释：

> **把这个童话意象抽掉，它承载的那句成人认知/哲学还站得住吗？**
> - 如果站得住 → 意象是**诗意的载体**，保留
> - 如果只剩可爱 → 意象是**儿童化稀释**，重写

《小王子》之所以被成人奉为经典，是因为它的童话意象都承载了只有成人才能真正咀嚼的重量——玫瑰不只是花（是"对独一无二之物负起责任"的隐喻），狐狸不只是狐狸（是"驯养与被驯养的联结本质"），星球不只是星球（是"每个人都活在自己微型宇宙里的孤独"）。童话形式没有稀释这些思想，**反而因为找到了正确的距离让它们更直抵人心**。

同类参照：Studio Ghibli（宫崎骏）· Italo Calvino（隐形的城市 / 宇宙奇趣全集）· Jorge Luis Borges · Winnie the Pooh 原著 · Maurice Sendak（Where the Wild Things Are）。这些作品都用童话外壳承载成人哲学，构成一个合法的、有悠久传统的艺术范式。

**测试案例**：
- ✅ M01 齿轮+碎片 → 承载"范畴化哲学"（通过）
- ✅ M10 花园自由种植 → 承载"审美即判据"（通过）
- ✅ 修复动画"花粉漩涡升起" → 承载"部分如何涌现为整体的美学体验"（通过）
- ❌（已废）"把混乱的朋友们送回各自颜色的家啦" → 抽掉童话只剩可爱（不通过，已在 v1.2 重写）
- ❌（待警惕）"全家欢乐大合唱"类修复动画 → 纯可爱无重量（若出现要重写）

### 4.3 角色设计

**莱米（主角）**
- 小兔子造型保持**简洁、稚拙、略带忧郁的童话感**——扁圆身形、细腿、小黑点眼、长耳，但不过度卖萌
- **线条与世界完全一致**：轮廓必须是清晰可见的手绘墨线，不能单独走柔软水彩角色路线
- **配色克制**：身体可用陶土红 + 灰绿 / 灰蓝，局部点一点赭石黄；颜色轻，不覆盖线条
- 服装和道具尽量简化，只保留飞行员身份所需的最少信息，不用多余徽章和装饰件
- **动画**：Dragon Bones / Cocos Skeleton2D 仍可用，但贴图表现应以线条轮廓 + 淡彩点染为主
- **情绪传达**：依赖姿态、停顿、比例和留白，不依赖夸张面部表情
- **尺度规则**：莱米通常比主装置小很多，主要承担情感锚点和尺度参照的作用

**流辉美慧号（飞船）**
- 飞船同样采用**简约手绘机械语言**：清楚的外轮廓、少量必要功能部件、极少装饰
- 可保留黄铜暖色作为关键机械识别，但只能点在舷窗边框、吸盘接口、铰链等必要结构上
- 吸盘、透镜、喷口等工具要像真正可用的小型工作器具，而不是复杂机械秀场
- 每阶段升级时优先增加**功能性变化**，而不是叠加越来越多的装饰附件

**星星（关卡容器）**
- 每颗星星首先是**一个功能明确的简约装置**，其次才是诗意物件
- 轮廓用清晰墨线勾出，结构尽量单纯：一个主装置 + 少量必要部件 + 少量故障线索
- 故障状态通过**卡住、错位、少量碎片、线条断裂、局部失色**来表现，不靠大面积华丽损坏
- 修复后不是变得更花，而是**结构恢复清楚、线条更连贯、颜色轻微回暖**
- 可保留"发光"概念，但光应很克制，只像纸面上被轻轻擦亮，而不是强特效

### 4.4 UI 设计

- **按钮 / 边框 / HUD 全部走清晰手绘墨线**，线条不能因为 UI 扁平化而消失
- UI 用色同样克制：米白、灰蓝、灰绿为底，陶土红 / 赭石黄只做轻微状态区分
- **星图地图**：以留白和线条组织信息，已完成星星只做轻微回暖，不做整颗高亮饱和发光
- **智慧日志本**：像旧纸上的手工记录本，允许一点纸张质感和局部点色，但仍以线条为主
- **认知工具册**（新增机制，详见 §6.5）：card 正面以手绘线稿 + 淡彩剪影表达，不做厚涂插画卡面
- **HUD 极简**：仅显示当前关卡名、提示按钮、暂停按钮；宁可空，也不要加装饰

### 5.2 M01：记忆齿轮的卡顿

**认知目标**：分类与归纳

**场景描述**：
一颗巨大的齿轮状星星，齿槽里塞满五彩斑斓、形状各异的"记忆碎片"。齿轮侧面有三个空着的彩色插槽（红、蓝、黄）。远处漂浮着对应的过滤器原型。齿轮因卡死而静默，星光暗淡。

**视觉要素**：
- 金属质感齿轮，暗淡无光
- 五彩碎片散落在齿槽中（3 种颜色 × 3 种形状 = 9 类，每类 2-3 个，共 ~24 个碎片）
- 三个发光的彩色过滤器浮在远处（红 / 蓝 / 黄）
- 中央收纳盘含 9 个形状吸附槽（圆 / 三角 / 六边 × 3 色），每槽形状+颜色双维度匹配

**交互流程**：
1. 拖拽红色过滤器 → 插入插槽 → 非红色碎片变暗（`color-filter` shader）
2. 用飞船吸盘（长按拖拽）吸取闪亮的红色碎片 → 拖到收纳盘
3. 同形状碎片自动叠放（`SnapZone` 吸附）
4. 更换过滤器，重复流程
5. 全部分类完成 → 修复动画

**使用的组件**：`DragHandler`, `SnapZone`, `FilterSystem`
**使用的 Shader**：`star-glow`, `color-filter`, `particle-flow`

**胜利条件**：`{ type: "all_sorted", params: { dimensions: ["color", "shape"], colors: ["red", "blue", "yellow"], shapes: ["circle", "triangle", "hexagon"] } }` — 所有 24 片碎片按"颜色 + 形状"落入正确的 9 类槽位才判成功（双维度交叉分类）

**修复动画**：齿轮转动 → 碎片以漩涡状喷出 → 化为持续星光 → 镜头拉远展示修复后的星星

**智慧结晶**："秩序，是为相似之物找到归处。"

### 5.3 M02：迷失的导航罗盘

**认知目标**：系统校准与参照 — "我的尺子准吗？"

**场景描述**：
古老观测台内部，中央是多层嵌套的青铜星象罗盘，各层圆环刻着符号（日、月、云、箭），彼此错位。漂浮着发光实物模型：小太阳、小月亮、云、风向标。

**视觉要素**：
- 青铜质感的嵌套罗盘，符号微弱发光
- 漂浮的实物模型作为校准基准
- 校准完成后投射三维星航道全息图

**交互流程**：
1. 抓取小太阳模型 → 靠近罗盘 → 太阳符号环自动旋转对齐
2. 同理校准月亮、云
3. 用飞船引擎向风向标吹气 → 旋转指向
4. 根据指向手动旋转最外层空白环
5. 所有环校准 → 投射三维星航道图

**使用的组件**：`DragHandler`, `SnapZone`, `RotateHandler`
**使用的 Shader**：`star-glow`, `hologram`

**胜利条件**：`{ type: "alignment", params: { rings: ["sun", "moon", "cloud", "outer"], tolerance: 5 } }`

**修复动画**：罗盘各环同时旋转加速 → 中央投射出三维星航道 → 星光沿航道流动

**智慧结晶**："先对准真正的参照，尺子才会开始说真话。"

### 5.4 M03：冻结的星之河

**认知目标**：间接策略与资源管理 — "除了硬上，还能怎么做？"

**场景描述**：
星星表面冻结的"光之液"河道，矗立着不透明的"情绪浮冰"。飞船有聚光透镜。河床下有导流板控制开关。

**视觉要素**：
- 冰蓝色的冻结河道，表面有裂纹纹理
- 半透明浮冰内部封存着彩色气泡——每块冰就是一团被冻住的"情绪"：
  - 红气泡 = **激动** / **愤怒**（融化释放后使河水激荡、冲垮下游）
  - 蓝气泡 = **恐惧** / **退缩**（融化释放后使局部水流重新凝结）
  - 无色气泡 = **平静**（安全，可直接开路）
- 融化时流体粒子效果（`fluid-sim` shader）

**交互流程**：
1. 聚光透镜照射浮冰 → 融化并释放内部情绪气体
2. 观察每块冰的气泡颜色，判断它适合"正面处理"还是"绕开"——强攻红冰会反噬，正面冻蓝冰会停摆
3. 策略：先融无色/平静的冰 → 打开河道段，不与强烈情绪正面对抗
4. 对于关键路径上的红/蓝情绪冰，操作导流板把融水绕过去，或用另一种情绪气体中和
5. 河道贯通 → 光之液流动

**使用的组件**：`DragHandler`（透镜方向）, `SliderControl`（导流板）
**使用的 Shader**：`fluid-sim`, `star-glow`, `particle-flow`

**胜利条件**：`{ type: "threshold", params: { metric: "river_flow_rate", target: 0.8 } }`

**修复动画**：河道全线贯通 → 光之液以加速流动冲破最后冰层 → 河面升起暖色光雾

**智慧结晶**："硬物未必以硬破之——借流动者，亦可移山。"

### 5.5 M04：纠缠的星际信号

**认知目标**：多重条件同时满足 — "怎样同时满足所有要求？"

**场景描述**：
通讯塔星星，多个频率的信号纠缠成噪音。有频率旋钮、带通滤波器和信号路由板。每个信号必须同时满足：正确频率 + 正确带宽 + 正确路由目标。

**视觉要素**：
- 高耸的通讯塔，天线阵列闪烁
- 可视化频谱面板，不同信号以不同颜色显示
- 信号路由板上的连线发光

**交互流程**：
1. 旋转频率旋钮 → 可视化频谱中不同信号的峰值
2. 调节带通滤波器宽度 → 分离出单一信号
3. 将分离的信号路由到对应的接收天线
4. 挑战：调整一个信号的频率可能影响相邻信号（多条件交叉约束）
5. 所有信号同时满足频率+带宽+路由三个条件 → 通讯恢复

**使用的组件**：`RotateHandler`, `SliderControl`, `Connector`
**使用的 Shader**：`wave-propagation`, `star-glow`

**胜利条件**：`{ type: "all_conditions_met", params: { signals: ["alpha", "beta", "gamma"], conditions: ["frequency", "bandwidth", "routing"] } }`

**修复动画**：三束信号光柱从天线射向星空 → 汇聚成一个通讯节点 → 通讯节点绽放星光

**智慧结晶**："解不在任一条件上，而在它们共同穿过的那一点。"

### 5.6 M05：断裂的平衡星盘

**认知目标**：动态平衡与系统调节 — "怎样让它不倒也不僵？"

**场景描述**：
巨大天平星星，两端悬挂着不同重量和颜色的星石。天平严重倾斜，核心轴承因不平衡而发热变红。平衡不是静止的——风会周期性吹动天平。

**视觉要素**：
- 金属质感天平，轴承根据平衡度变色（红→黄→蓝）
- 星石有不同大小和颜色（大小 = 基础重量，颜色 = 共鸣属性）
- **同色星石之间有淡淡的脉动磁感线**——提示"颜色会相互拉扯，产生额外重力"，让玩家在第一次观察时就能看出这条规则（而非试错）
- 周期性的星风粒子吹过天平，风向和风力有箭头微弱可视化

**交互流程**：
1. 观察星石的重量标记和颜色
2. 拖拽星石在两端之间移动
3. 需要考虑颜色组合规则（同色相邻会产生额外重力）
4. 星风周期性吹来，打破静态平衡 → 需要预判并提前调整
5. 维持动态平衡持续 5 秒 → 轴承冷却，天平稳定

**使用的组件**：`DragHandler`, `SnapZone`
**使用的 Shader**：`star-glow`, `color-filter`, `particle-flow`

**胜利条件**：`{ type: "dynamic_balance", params: { metric: "balance_delta", tolerance: 0.1, sustainSeconds: 5 } }`

**修复动画**：天平两端同时发光 → 星石升起围绕轴心旋转 → 化为一个稳定的光之陀螺

**智慧结晶**："稳定不是静止，而是无数次及时的微小回应。"

### 5.7 M06：休眠的因果链

**认知目标**：因果排序与依赖推理 — "谁必须在谁前面？"

**场景描述**：
星星内部是一条断裂的"因果链" — 一系列齿轮、杠杆、管道组成的鲁布·戈德堡机械。几个关键节点断开或缺失零件。

**视觉要素**：
- 蒸汽朋克风格的机械链条
- 断裂处有火花闪烁
- 零件箱悬浮在一侧

**交互流程**：
1. 观察机械链条的走向和断裂点
2. 从零件箱中选取正确零件填补断裂处
3. 连接零件时需考虑传动方向（齿轮旋转方向）和因果依赖（A 必须在 B 之前）
4. 激活起点 → 链式反应沿正确路径传递 → 终点亮起

**使用的组件**：`DragHandler`, `SnapZone`, `Connector`
**使用的 Shader**：`star-glow`, `particle-flow`

**胜利条件**：`{ type: "causal_chain", params: { chain: ["start", "gear_1", "lever_2", "pipe_3", "end"], orderMatters: true } }`

**修复动画**：链条全线启动 → 多米诺骨牌般逐级传递 → 终点爆发星光

**智慧结晶**："链条不讲平等——谁先谁后，决定它是否成立。"

### 5.8 M07：碎裂的星之万花筒

**认知目标**：分解与重组 — "这个大问题能拆成哪几个小问题？"

**场景描述**：
一颗由精密棱镜组成的万花筒星星，整体裂成 4-5 块大碎片，每块碎片内部的小棱镜仍然完好，但碎片之间的连接断裂。碎片漂浮在星星原来的位置附近，微微旋转。

**视觉要素**：
- 半透明的棱镜碎片，内部折射彩虹光
- 碎片之间有隐约的"断裂线"闪烁
- 中央有一个空的"核心框架"骨架
- 修复后棱镜旋转投射出完整的光之万花筒

**交互流程**：
1. 观察碎片 → 点击放大查看内部棱镜排列（理解每块的结构）
2. 识别核心框架的槽位形状 → 与碎片轮廓匹配
3. 逐块拖拽碎片到框架 → SnapZone 吸附（"拆大为小"的逆过程）
4. 每块就位后，调整碎片旋转角度使内部棱镜的折射面对齐（"重组"的精细调整）
5. 全部对齐 → 万花筒旋转，投射出完整的星光图案

**使用的组件**：`DragHandler`, `SnapZone`, `RotateHandler`
**使用的 Shader**：`star-glow`, `color-filter`, `particle-flow`

**胜利条件**：`{ type: "assembly", params: { pieces: ["frag_A", "frag_B", "frag_C", "frag_D"], alignmentTolerance: 8 } }`

**修复动画**：万花筒高速旋转 → 棱镜折射出万道彩光 → 光柱射向星空形成彩虹星云

**智慧结晶**："庞然之物皆可解为微小——重建之时，接缝即其灵魂。"

### 5.9 M08：倒流的星之瀑布

**认知目标**：逆向推理 — "要到达终点，上一步必须是什么？"

**场景描述**：
一颗瀑布星星，星光之水从顶端应该流向底部的"生命池"，但瀑布干涸了。从底部的生命池开始，有一条倒流的光痕隐约向上延伸，暗示水曾经流过的路径。路径上有多个分叉和阀门节点。

**视觉要素**：
- 干涸的阶梯式瀑布岩壁
- 底部微弱发光的生命池
- 从底部向上的淡蓝色光痕"路径提示"
- 路径节点处有可切换的闸门

**交互流程**：
1. 从底部（终点）开始 → 点击生命池激活逆向追踪模式
2. 光痕从池子向上延伸到第一个分叉 → 选择正确的上游方向（哪条路能到水源？）
3. 每到一个阀门节点 → 需要判断"水要从这里流过，阀门应该朝哪个方向？" → 旋转阀门
4. 逐级向上追溯，每步都是"要让水到达这里，前一步必须怎样？"
5. 到达顶部水源 → 打开总闸 → 水沿着你开辟的路径正向流下 → 瀑布复活

**使用的组件**：`DragHandler`, `RotateHandler`, `Connector`
**使用的 Shader**：`fluid-sim`, `star-glow`, `particle-flow`

**胜利条件**：`{ type: "path_reverse", params: { start: "source_top", end: "pool_bottom", valves: ["v1", "v2", "v3", "v4"], mode: "reverse_then_verify" } }` — 先从终点逆向确定阀门朝向，再由源头放水做一次正向验证才算通过

**修复动画**：水从顶端倾泻而下 → 经过每个正确开启的阀门 → 瀑布全线复活，彩虹升起

**智慧结晶**："正向寻路失败时，便从终点倒走——每一步只问：上一步一定是什么。"

### 5.10 M09：折叠的星之信封

**认知目标**：空间推理与心理旋转 — "转过来/翻过去会变成什么样？"

**场景描述**：
星星表面是一张巨大的"星之信封"—— 一块展开的平面金属板，上面刻着星座图案的碎片。需要沿着折痕把平面折叠成立体的星之盒子，折叠后碎片图案恰好拼成完整的星座。

**视觉要素**：
- 展平的**十字形金属板，共 6 个正方形面板**（中心 1 个为底，上下左右各 1 个对应盒子四侧，最上端再接 1 个对应盒盖），对应立方体的 6 面
- 每个面板上刻着部分星座线条和星点，只有正确折叠后相邻面板的线条才能连成完整星座
- 折痕处有虚线发光提示，并标注折叠方向（山折 / 谷折）
- 折叠时金属板有机械铰链动画
- 完成后立体盒子旋转展示完整星座

**交互流程**：
1. 俯视展开图 → 观察各面板上的星座碎片
2. 需要在脑中想象"折起来之后哪些边会贴合"
3. 拖拽面板沿折痕翻转（滑动手势控制折叠角度）
4. 折叠不对 → 星座线条不对齐，视觉反馈（线条错位闪红）
5. 可以展开重试 → 尝试不同的折叠顺序
6. 正确折叠 → 六面星座完美对齐 → 盒子发光，星座从表面升起成为 3D 全息投影

**使用的组件**：`DragHandler`, `RotateHandler`
**使用的 Shader**：`star-glow`, `hologram`, `particle-flow`

**胜利条件**：`{ type: "assembly", params: { faces: ["top", "bottom", "left", "right", "front", "back"], alignmentRequired: true } }`

**修复动画**：盒子缓缓旋转 → 六面星座光线从盒中射出 → 形成环绕的立体星图

**智慧结晶**："二维之中藏着三维的许诺——只需想象那两条边相遇的一刻。"

### 5.11 M10：沉睡的星之花园

**认知目标**：探索、创造与开放式解决 — "我觉得什么是好的？"

**场景描述**：
一颗曾经繁茂的花园星星，现在是一片空旷的沃土。土壤、水源、光照都是充足的，但没有任何植物。飞船的工具箱里有各种"星之种子"（不同颜色、形状、大小），没有"正确答案"—— 玩家自由创造自己的花园。

**视觉要素**：
- 圆形花园地块，分为若干种植区
- 左侧悬浮的种子工具箱（8-12 种不同的星之种子）
- 种下后植物以加速动画生长，不同种子组合产生不同的视觉效果
- 阳光/水/风的环境粒子

**交互流程**：
1. 自由选择种子 → 拖拽到任意种植区
2. 种子种下后即刻生长 → 观察效果
3. 相邻种子会产生"共鸣效果"（颜色混合、藤蔓互连、光芒交织），鼓励实验
4. 可以拔掉重种（长按移除）
5. 没有失败条件 → 当玩家觉得满意时，按下"绽放"按钮
6. 花园全景展示 → 生成独特的"花园之印"图案（每个玩家不同）

**使用的组件**：`DragHandler`, `SnapZone`
**使用的 Shader**：`star-glow`, `particle-flow`, `color-filter`

**胜利条件**：`{ type: "creative_threshold", params: { minPlantsPlaced: 5, playerConfirm: true } }`

**修复动画**：花园全部植物同时绽放 → 花粉形成漩涡上升 → 化为星之花园的独特光印

**智慧结晶**："当没有正确答案，审美便成为判据——此刻你是作者，而非解题者。"

## 六、游戏流程与 UI

### 6.1 场景流转

```
Boot (加载) → Home (星图地图) → Puzzle (谜题) → Home (返回地图)
                                      ↕
                               Journal (智慧日志)
```

### 6.2 星图地图（Home）

- 俯视星空视角，10 颗星星散布（第一阶段）
- 已完成的星星发出明亮星光，连线形成星座雏形
- 点击可用星星 → 进入谜题
- 锁定的星星显示暗淡 + 锁图标
- 底部导航栏：日志本、设置、飞船信息

### 6.3 谜题场景（Puzzle）

- 顶部：关卡名称（可收起）
- 右上：暂停按钮、提示按钮（带冷却指示）
- 左下：莱米头像 + 对话气泡（引导文字）
- 中央：谜题交互区域
- 修复成功后：全屏修复动画 → 智慧结晶弹出 → 评价（1-3星）→ 返回地图

### 6.4 提示系统

- 3 级提示，逐级解锁（30秒/60秒/90秒无操作后）
- 第 1 级：轻微方向暗示（如某个区域微微发光）
- 第 2 级：文字提示 + 关键物体高亮
- 第 3 级：操作步骤提示 + 箭头引导
- 使用提示越多，获得的星评越少

### 6.5 智慧日志本 & 认知工具册（Journal & Toolkit）

这是游戏的"**玩家能带走的礼物**"——也是"修复即提炼"定位的**核心落地载体**。它不是一个装饰性的成就系统，而是整个游戏的**可带出产物**：玩家离开游戏时手中握着一本 33 张 index card 的认知工具册，卡上的工具和场景是他自己玩过、体验过、命名过的。

#### 6.5.1 数据结构

每关通关时，玩家获得一张**认知工具卡（ToolCard）**，数据结构如下：

```typescript
interface ToolCard {
  puzzleId: string;            // "m01"
  stage: number;               // 1-5

  // 正面（感性 / 情感面）
  front: {
    toolName: string;          // "分类与归纳"
    scene: string;             // 场景剪影缩略图（清晰墨线 + 低饱和淡彩点染）
    wisdomCrystal: string;     // "秩序，是为相似之物找到归处。"
  };

  // 反面（理性 / 使用面）
  back: {
    coreAction: string;        // 一句话描述这个工具的核心动作
                               // 例："在杂乱事物中找到共同属性，按属性归组"
    whenToUse: string[];       // 3-5 个"什么时候该想起这张卡"
                               // 例：["整理文件时" / "拆书本笔记时" / "生物分类作业时"]
    realLifeExamples: string[]; // 2-3 个生活类比
                               // 例：["把冰箱食物按品类归位" / "衣柜按季节分层"]
    commonTraps: string;       // 使用该工具时容易掉进的坑
                               // 例："分类维度选错：'按颜色'vs'按形状'可能都对，看目的"
  };

  unlockedAt: number;          // timestamp
}
```

#### 6.5.2 UI 结构：一本书、两种翻法

玩家的"智慧本"有两种切换视图：

**日志视图（Journal Mode）** — 情感面 / 按时间
- 翻页式 UI，每页一张卡的**正面**
- 水彩纸底 + 墨线手写体文字 + 谜题剪影
- 按通关时间顺序排列（"我的旅程"叙事感）
- 适合回顾"我走过这里"的情绪

**工具册视图（Toolkit Mode）** — 理性面 / 按工具
- 卡片式索引 UI，每张卡可翻转看**反面**
- 按 5 阶段分章节，或按"动作类型"标签（分类类 / 逆向类 / 系统类 / 反观类 / 生成类）筛选
- 支持**搜索**："我当下这个问题，哪张卡能帮我？"
- 适合查阅"我该用哪个工具"

#### 6.5.3 反面内容的生成

反面的 `whenToUse` / `realLifeExamples` / `commonTraps` 这些"可调用"内容需要精心写作。写作准则：

- **不说教**：不写"你应该……"，而是"Ta 会在这里出现"——场景化唤起
- **生活化**：例子必须是成人日常会遇到的（整理、沟通、决策、学习、创作），不是课本场景
- **可搜索**：whenToUse 包含搜索关键词（"delay"、"决策"、"卡住"等），方便玩家按当下问题查找
- **诚实**：commonTraps 写真实的失败模式，不美化工具

**示例 · M01 分类卡反面**：
- 核心动作：在杂乱事物中找到共同属性，按属性归组
- 何时使用：整理一堆笔记不知从何下手时 / 需要把大量信息压缩总结时 / 面对多个选项想不清它们的关系时
- 生活例子：整理书架（按主题 vs 按颜色 vs 按作者）/ 做年度复盘（按月 vs 按项目 vs 按情绪）
- 常见陷阱：分类维度选错——同一堆东西用"颜色分"和"用途分"得到两种完全不同的结构，关键是"分类是为了什么目的"

#### 6.5.4 技术与美术约束

- 每张卡的美术 = 正面墨线剪影 + 点彩色 ≈ 一张邮票级的缩略插画
- 33 张卡的设计工作量 ≈ 33 张小插画（比 33 个关卡场景便宜 80%）
- 支持**导出 PNG 分享**（"我收藏了我的第一张思维工具卡：M01 分类"——自然传播载体）
- 长期：**导出为 Anki 卡包 / PDF 小册子**，让工具真正进入玩家的日常复盘流程

#### 6.5.5 为什么这是"修复即提炼"的核心机制

这个设计直接回应了 Codex 两轮审阅中最狠的那条"**概念融合 / 范式转换这些哲学工具根本做不成交互**"——

> 如果目的不是"从零教会玩家什么是概念融合"，而是"**让玩家玩了 M30 熔炉之后，获得一张叫'概念融合'的卡，反面写着 Ta 生活里已经做过这件事的 3 个例子**"，那游戏就不再是"哲学教科书"，而是"**认知工具的命名仪式**"。

玩家 离开游戏 ≠ 离开这些工具——卡片随时可查，可分享，可在生活中被调用。**这才是"不看回本"的 passion project 真正的"回报"形式：它让作者的思想进入别人的生活。**

## 十、技术风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|---------|
| Shader 性能在低端设备差 | 视觉效果降级 | 实现 LOD 分级，低配设备关闭复杂效果 |
| 33 关配置维护复杂 | 开发效率 | 关卡编辑器工具（后期） |
| 微信小游戏包体限制 + 成人定位错位 | 无法发布 / 受众偏移 | 重新评估是否发布该渠道；资源分包首包 < 4MB 作兜底 |
| AI 生成美术风格不一致 | 视觉割裂 | 统一以“线条优先、低饱和淡彩、功能性简约机械”为审核硬标准；提示词与参考图全部围绕同一套手绘线条语言收敛 |
| 后期关卡（M24-M33）逻辑复杂 | 开发周期长 | Stage 5 先做危险原型验证，失败则砍关；核心数学/物理模块与渲染层解耦 |
| **Stage 5 "命名仪式"可能是空壳盖章**（Codex Round 3 诊断） | 游戏核心体验坍塌 | P1-b 必做 Stage 5 原型验证；若 M30 原型（Stage 5 最硬试金石）无法让玩家真实经历该认知动作，Stage 5 整体放弃或重构 |
| **认知工具册可能反噬游戏本体** | 玩家把卡片当经验值集邮 | ToolCard 解锁条件与关卡真实完成强绑定，禁止通过跳关/作弊获得；Anki 导出放在完成 50% 关卡后才解锁 |
| **反面文案废话工厂风险** | 工具册成为 AI 味注释系统 | 反面文案不得由 AI 批量生成，每条必须人工过《小王子》测试法（§4.2），量足即失败 |
