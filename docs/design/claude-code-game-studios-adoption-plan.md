# Claude Code Game Studios 可迁移机制清单

> 类型: 内部执行建议 | 日期: 2026-03-24 | 目标项目: 《流辉美慧号：星图守护者》

---

## 一句话结论

`Claude Code Game Studios` 不适合整套搬进来，但其中有几类机制非常值得迁移:

1. **流程护栏**: 让 AI 不容易"写着写着就跑偏"
2. **状态外置**: 让会话中断后还能续上
3. **结构化模板**: 让设计与实现文档不再散乱
4. **轻量协作编排**: 只在真的需要时才启用多 agent

对《星图守护者》来说，最佳策略不是"建一个 48 人 AI 工作室"，而是:

**抽取 20% 的机制，解决 80% 的协作混乱。**

---

## 迁移原则

所有迁移都应满足下面 4 条，否则宁可不迁:

1. **不依赖 Godot / Unity / UE5 专用约定**
2. **适配 Cocos Creator 3.8 + TypeScript**
3. **服务于 1-2 人 + AI 的小团队现实**
4. **优先降低混乱，不优先增加仪式感**

---

## 建议立即迁移的 4 项

### 1. 资源校验 Hook

**迁移动机**: 你们的项目是强资源驱动型。关卡 JSON、图集、shader、占位资源一旦命名混乱，后期会非常痛苦。

**建议迁移内容**:

- 增加一个轻量版 `validate-assets` hook
- 校验资源命名规范
- 校验配置 JSON 是否合法
- 校验关键目录下是否出现明显错放文件

**适配到本项目后的检查重点**:

- `resources/configs/` 下 JSON 文件命名与结构
- `scripts/levels/stageN/` 与关卡编号的一致性
- `scripts/interaction/` 里是否混入关卡私有逻辑
- `shaders/` 命名是否可追踪

**为什么优先级最高**:

这是最便宜、最稳定、最不依赖人记忆的一种约束方式。

---

### 2. 文件即状态

**迁移动机**: 这个项目设计文档多、关卡多、上下文链长，最容易丢的是"现在推进到哪一步"。

**建议迁移内容**:

- 在 `production/` 下维护一个简单状态文件
- 记录当前阶段、当前主任务、最近完成项、阻塞项、下一步
- 不追求复杂，只要足够让下一次会话快速接手

**推荐最小结构**:

- `production/active.md`: 当前主线工作
- `production/stage.txt`: 当前所处阶段
- 可选 `production/notes.md`: 临时遗留问题

**为什么值得做**:

这会显著减少"重新解释一遍项目现在做到哪"的时间浪费。

---

### 3. 状态线 / 阶段 Breadcrumb

**迁移动机**: Game Studios 里最容易被低估的不是 agent 数，而是它把项目状态做成了可见的。

**建议迁移内容**:

- 不一定要做终端脚本版 `statusline.sh`
- 但至少应该有一个统一的阶段标记方式
- 例如把工作流粗分成:
  - Vision
  - Toolset
  - Stage 1 Design
  - Stage 1 Implementation
  - Polish
  - Content Expansion

**在本项目里的价值**:

- 帮你区分"我们现在是在改设计，还是在做实现"
- 帮 AI 避免把未拍板设计当成已定稿规格
- 帮后续多线程推进时保持同一上下文

---

### 4. 模板体系扩充

你们现在已经有设计文档基础，但还缺几种非常实用的中间层模板。

**建议新增 3 类模板**:

- `关卡实现模板`
  - 适合把设计规格转成实际开发任务
  - 包括: 关卡目标、交互组件、配置项、失败态、提示策略、验收标准
- `交互组件模板`
  - 用于 `scripts/interaction/` 里的可复用组件
  - 包括: 组件职责、输入输出、可配置参数、复用边界、依赖资源
- `Shader 设计模板`
  - 用于 `shaders/` 目录的效果设计与性能约束
  - 包括: 视觉目标、参数、平台限制、降级方案、复用场景

**为什么这项重要**:

你们不是缺"创意"，而是缺"从设计到实现的稳定过渡层"。

---

## 建议中期迁移的 3 项

### 5. 轻量 Agent 审计

如果后面真的开始频繁并行使用多个 agent，建议记录:

- 谁做了什么
- 改了哪些文件
- 哪个任务依赖哪个任务

不需要完整复刻 `log-agent.sh`，但至少要避免后期出现:
"这是谁改的？为什么改？现在还能不能继续改？"

---

### 6. 设计系统映射工作流

Game Studios 的 `/map-systems` 很值得借，但不必照搬成命令。

你们可以把它改造成一份项目内固定流程:

- 先列系统清单
- 再列依赖关系
- 再列设计顺序
- 最后再列实现顺序

对《星图守护者》尤其有用，因为你们不是 50 个完全独立的小题，而是一套可复用工具、组件、配置和提示系统。

---

### 7. 分章节协作文档法

当 `game-design-spec.md` 继续膨胀时，建议借用 `/design-system` 背后的方法:

- 先搭章节骨架
- 一章一章拍板
- 每章写完立刻落盘

这对长规格文档很关键，因为它可以减少一次性重写大文档造成的失控和反复。

---

## 不建议迁移的部分

### 1. 48 Agent 分层架构

对你们当前规模完全过重。它解决的是"大型协作秩序问题"，不是"小团队设计清晰度问题"。

### 2. 引擎专用专家集

Godot / Unity / UE5 的 specialist 对当前项目几乎没有直接价值，反而会制造错误的实现假设。

### 3. 过于严格的 Question-First 仪式

它的协作理念值得借，但如果每次写文件都先走完整提问批准流程，会拖慢日常推进。  
更适合你们的方式是:

- 重大结构改动先拍板
- 小型实现和文档整理允许先做后汇报

---

## 适合《星图守护者》的精简版落地方案

如果只做一个最小可行迁移包，我建议就是下面这 5 件:

1. `production/active.md`
2. `production/stage.txt`
3. 资源校验 hook
4. `关卡实现模板`
5. `交互组件模板`

这 5 件加起来，已经足够把"长设计链条 + AI 协作"的混乱降下来，而且不会把项目搞成流程过重。

---

## 建议实施顺序

### 第 1 波: 立刻做

- 资源校验 hook
- `production/active.md`
- `production/stage.txt`

### 第 2 波: 设计与实现衔接期做

- 关卡实现模板
- 交互组件模板
- Shader 设计模板

### 第 3 波: 项目体量上来后再做

- agent 审计
- 状态线
- 系统映射工作流

---

## 最终判断

对《星图守护者》来说，`Claude Code Game Studios` 最值得学的，不是"多 agent 扮演工作室"，而是下面这件事:

**把 AI 协作从"靠聊天记忆"升级成"靠文件、规则、状态和模板运作"。**

这才是它真正可迁移、也最适合你们当前阶段的部分。
